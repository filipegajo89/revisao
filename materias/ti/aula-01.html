BANCO DE DADOS - Conceitos Fundamentais e Modelagem

Um Banco de Dados é um conjunto organizado de dados relacionados, gerenciado por um Sistema de Gerenciamento de Banco de Dados (SGBD).

SGBD (SISTEMA DE GERENCIAMENTO DE BANCO DE DADOS):
- Software que gerencia acesso, integridade e consistência dos dados
- Garante segurança, disponibilidade e recuperação
- Exemplos: Oracle, SQL Server, MySQL, PostgreSQL

MODELO RELACIONAL:
- Dados organizados em tabelas (relações)
- Tabelas contêm linhas (tuplas/registros) e colunas (atributos/campos)
- Relações entre tabelas através de chaves estrangeiras

CHAVES:
- Chave Primária (PK): Identificador único do registro. Não pode ser nula.
- Chave Estrangeira (FK): Referencia PK de outra tabela. Mantém integridade referencial.
- Chave Candidata: Qualquer coluna que poderia ser PK

SQL (STRUCTURED QUERY LANGUAGE):
- DDL (Data Definition Language): CREATE, ALTER, DROP - definem estrutura
- DML (Data Manipulation Language): SELECT, INSERT, UPDATE, DELETE - manipulam dados
- DCL (Data Control Language): GRANT, REVOKE - controlam acesso

NORMALIZAÇÃO:
- 1FN: Elimina atributos multivaluados
- 2FN: Elimina dependências parciais (colunas não-chave dependem só da PK)
- 3FN: Elimina dependências transitivas (colunas não-chave dependem umas das outras)
- BCNF: Forma normal de Boyce-Codd

INTEGRIDADE REFERENCIAL:
- Garante que valores de FK existem como PK na tabela referenciada
- Usa restrições: ON DELETE/UPDATE CASCADE, RESTRICT, SET NULL

TRANSAÇÕES E ACID:
- Atomicidade: Tudo ou nada (commitada ou desfeita)
- Consistência: Respeita todas as restrições
- Isolamento: Transações não interferem umas nas outras
- Durabilidade: Dados persisted após commit

Na verdade, as estruturas para armazenamento de dados e os métodos de acesso ao banco de dados
fazem parte do projeto físico de um banco de dados, não do projeto lógico.
---
Perfeito! A atomicidade desempenha um papel crucial nas transações de banco de dados relacional,
garantindo que os dados sejam precisos, consistentes e estejam em conformidade com as regras e
políticas da empresa.

Diego, por que não pode ser Consistência? Tanto atomicidade quanto a consistência tratam da
integridade dos dados. A frase realmente menciona que a propriedade garante que os dados estejam
precisos e sigam regras, regulamentos e políticas da empresa. No entanto, o contexto sugere que a
questão está enfatizando que a transação ocorra por inteiro ou não ocorra, o que caracteriza Atomicidade.
Se o foco fosse evitar que regras fossem quebradas (Ex: saldo negativo, chave estrangeira inválida), então
seria Consistência. Como a questão trata da transação ser executada por completo ou revertida, trata-se
de Atomicidade.
---
A propriedade envolvida quando uma transação de banco de dados é completada com sucesso e as
mudanças que ela fez no banco persistem, mesmo que existam falhas no sistema, é a durabilidade. Ela
assegura que, uma vez que uma transação seja confirmada (completada com sucesso) e suas mudanças
sejam gravadas no banco de dados, essas alterações sejam permanentes, mesmo que ocorram falhas
subsequentes, como falta de energia, falha de hardware ou problemas de software. Em outras palavras,
as alterações devem ser duráveis e não podem ser desfeitas.

CESGRANRIO
65. (CESGRANRIO / IPEA - 2024) Considere que, em um banco de dados preparado para persistir dados
de apoio à realização de estudos na área agropecuária, existem duas tabelas:

IMÓVEL (cod-sncr, denominacao, cod-mun, area-total)
MUNICÍPIO (cod-mun, uf, qtd-imoveis)

A chave primária de um imóvel é o seu código no Sistema Nacional de Cadastro Rural (SNCR), e a de
um município é o seu código, segundo o IBGE. A coluna IMÓVEL.cod-mun é chave estrangeira e
referencia MUNICÍPIO.cod-mun.

Uma aplicação, ou sistema, realiza regularmente a seguinte transação nesse banco de dados:

Início da transação
Inserção dos dados de um novo imóvel na tabela IMÓVEL
Atualização de qtd-imoveis na tabela MUNICÍPIO
Commit

Nesse exemplo, a propriedade de atomicidade de uma transação garantirá que:

a) a integridade referencial entre as duas tabelas sempre será mantida.
b) apenas usuários autorizados poderão consultar os novos dados inseridos.
c) nenhuma das duas tabelas será modificada em caso de problema durante a sua execução.
d) não existirão dois municípios com o mesmo código.
e) não haverá deadlock entre duas transações concorrentes.
