Teoria da Informação estabelece conceitos fundamentais para compreender o ciclo de processamento e significação de dados em contextos organizacionais e de tecnologia da informação.

DADOS: Elementos brutos, representados por símbolos ou valores sem significado intrínseco. São matéria-prima para informação. Exemplos: número "28", caractere "A", temperatura "25°C". Dados isolados não comunicam muito - precisam ser processados, contextualizados e interpretados para tornarem-se úteis. Diferem de informação porque informação é dados processados com significado.

INFORMAÇÃO: Resultado do processamento, organização, estruturação e análise de dados com objetivo de gerar significado útil para tomada de decisão. Dados transformados em contexto determinado. Exemplo: "28 colaboradores trabalham nesta filial" é informação (dado 28 + contexto = significado). Informação reduz incerteza e suporta processos decisórios. Deve ser: relevante (pertinente ao problema), oportuna (disponível no tempo certo), precisa (livre de erros), completa (suficiente para decisão).

CONHECIMENTO: Interpretação e compreensão da informação em contexto significativo. É aplicação da informação a problemas específicos. Exemplo: compreender que "28 colaboradores é insuficiente para produção desejada" é conhecimento (informação interpretada em contexto). Conhecimento permite prognósticos, tendências, correlações entre informações. Resulta em insights estratégicos.

SABEDORIA: Nível superior de conhecimento que incorpora julgamento, experiência e valores. Capacidade de aplicar conhecimento com discernimento em cenários novos. Saber QUANDO e COMO aplicar conhecimento adequadamente. Relacionada a experiência acumulada e compreensão profunda.

RELACIONAMENTO: Dados → Informação → Conhecimento → Sabedoria (hierarquia). Cada nível agrega mais interpretação, contexto e valor. Dados são brutos; informação tem significado; conhecimento tem padrões; sabedoria tem discernimento.

IMPORTÂNCIA PARA TI: Compreender essa hierarquia é fundamental para desenhar sistemas de informação que capturem dados (qual dado coletar?), processem em informações (qual transformação?), suportem conhecimento (qual análise?), e habilitarem sabedoria (qual decisão estratégica?).

QUESTÕES TÍPICAS CEBRASPE: Afirmações do tipo "(I) Dados são matéria-prima para informação. (II) Conhecimento é interpretação de informação em contexto significativo. (III) Sabedoria é aplicação de conhecimento com experiência." Todas verdadeiras. Banca explora compreensão profunda da distinção entre esses níveis.
Pegadinhas da banca
Confundir dados com informação: 'José, 32, técnico' são dados. 'José é técnico com 10 anos de experiência, logo é expert' é informação.
Conhecimento ≠ informação: conhecimento exige interpretação em contexto e padrões. Informação é dados contextualizados.
Sabedoria ≠ conhecimento: sabedoria incorpora julgamento e experiência em aplicação apropriada.
Pensar que informação dispensa contexto: mesmo valor numérico muda significado com contexto ('28 grama' vs '28 quilogramas').
Assumir que mais dados = melhor informação: sem processamento e estruturação apropriados, dados em volume pode confundir.

Ciclo de Vida de Dados descreve os estágios pelos quais dados passam desde sua criação até descarte ou arquivamento permanente em contextos organizacionais.

COLETA/CAPTURA: Primeira fase - obtenção de dados brutos de diversas fontes (sistemas transacionais, sensores, formulários, APIs externas). Dados são capturados em sua forma mais bruta, ainda sem processamento. Qualidade desde essa fase é crítica - "lixo entra, lixo sai" (GIGO - Garbage In, Garbage Out).

ARMAZENAMENTO: Dados capturados são armazenados em bancos de dados, data warehouses, data lakes ou sistemas de arquivo. Armazenamento deve ser seguro, escalável e acessível. Diferentes formatos (estruturado, semiestruturado, não-estruturado) requerem diferentes estratégias de armazenamento.

PROCESSAMENTO E TRANSFORMAÇÃO: Dados brutos são limpos, validados, transformados e estruturados para uso útil. Inclui ETL (Extract, Transform, Load): extrair dados de fontes diversas, transformar conforme regras de negócio, carregar em destino (data warehouse). Também inclui operações como deduplicação, normalização, agregação, enriquecimento.

ANÁLISE E INTERPRETAÇÃO: Dados processados são analisados para gerar insights. Inclui análise exploratória (entender dados), análise descritiva (o que ocorreu), análise preditiva (o que ocorrerá), análise prescritiva (o que fazer). Ferramentas de BI (Business Intelligence), data mining, machine learning aplicadas aqui.

DISSEMINAÇÃO/PUBLICAÇÃO: Resultados da análise são compartilhados com usuários finais através de relatórios, dashboards, APIs, publicações. Dados devem ser apresentados de forma compreensível e acionável.

ARQUIVAMENTO/RETENÇÃO: Dados históricos são retidos conforme políticas de negócio e regulamentações. Alguns dados são descartados (por lei, por falta de relevância), outros arquivados para auditoria e conformidade.

DESCARTE/DESTRUIÇÃO: Dados chegam ao fim de vida útil e são destruídos de forma segura (destruição física de mídia, deleção permanente de registros lógicos) para proteger privacidade e segurança.

ETL PROFUNDO: Extract (obtém dados de fonte - banco de dados, arquivos, APIs), Transform (aplica regras de negócio - validação, limpeza, cálculos, agregações, conversões de formato), Load (insere dados transformados em destino - data warehouse, data mart). Questões CEBRASPE/FCC exploram fases do ETL: "qual fase define como dados são carregados em data warehouse?" (Load). "Qual fase realiza modificações nos dados?" (Transform).
Pegadinhas da banca
Confundir ETL com ciclo completo: ETL é apenas uma parte (processamento) do ciclo de vida. Não inclui coleta inicial nem análise final.
Pensar que Load é onde dados entram primeiro: dados entram na Coleta, Load é saída do ETL.
Esquecer que Transform é onde qualidade é assegurada: limpeza, deduplicação, validação ocorrem nessa fase.
Confundir Data Warehouse com Data Lake: warehouse é estruturado e processado; lake é repositório de dados brutos em volume.
Assumir que dados são 'prontos' após ETL: análise adicional e enriquecimento podem ser necessários.

Metadados são 'dados sobre dados' - informações que descrevem características, propriedades, contexto e significado de outros dados. Sem metadados, dados brutos perdem significado.

METADADOS DESCRITIVOS: Informações que identificam e descrevem dados: título, autor, assunto, palavras-chave, descrição, data de criação, data de modificação, versão, idioma. Exemplo: documento com metadados "Relatório de Vendas Q2 2025, autor João Silva, criado 01/04/2025, atualizado 10/04/2025". Facilitam descoberta e compreensão.

METADADOS ESTRUTURAIS: Informações sobre organização e estrutura dos dados: como dados estão compostos (relacionamentos entre partes), formato de arquivo, esquema de banco de dados, definições de campos (tipo de dado, comprimento, restrições), índices. Exemplo: "Tabela Clientes possui campos ID (inteiro), Nome (texto 100 caracteres), Email (texto), Data_Cadastro (data)". Críticos para processamento correto.

METADADOS ADMINISTRATIVOS: Informações sobre gestão e controle de dados: direitos de acesso/permissões, políticas de retenção, proprietário, classificação de sensibilidade, conformidade regulatória, licenças de uso, histórico de auditoria. Exemplo: "Arquivo contém dados pessoais, acesso restrito a RH, retenção por 5 anos conforme LGPD, proprietário Departamento de Recursos Humanos".

APLICAÇÕES PRÁTICAS: (1) Pesquisa e Descoberta: metadados descritivos permitem buscar dados relevantes em repositórios grandes. (2) Integração de Dados: metadados estruturais definem como integrar dados de diferentes fontes. (3) Governança: metadados administrativos garantem conformidade, segurança, privacidade. (4) Qualidade de Dados: metadados ajudam validar integridade, completude, acurácia. (5) Documentação: metadados servem como documentação executável dos dados.

FERRAMENTAS E TECNOLOGIAS: Sistemas de gestão de metadados (metadata repositories), data catalogs (ferramentas de descoberta de dados), data lineage tools (rastreiam origem e transformações de dados). XML, JSON usados frequentemente para representar metadados estruturados.

EXEMPLOS EM DOCUMENTOS: Documentos Word/PDF contêm metadados como autor, data de criação, título, assunto, palavras-chave. Imagens JPEG contêm EXIF metadados: câmera, data da foto, localização GPS, iso, velocidade de obturador. Arquivos em geral contêm timestamps de criação/modificação, tamanho, permissões.
Pegadinhas da banca
Confundir metadados com dados: metadados descrevem dados (ex: 'altura é campo numérico de 3 dígitos'), dados são conteúdo ('175 cm').
Pensar que metadados descritivos bastam: sem metadados estruturais, dados não são processáveis por sistemas.
Metadados não garante qualidade: metadados administrativos indicam políticas, mas não forçam conformidade.
Esquecer que metadados também exigem manutenção: metadados desatualizados podem ser piores que ausência.
Confundir metadata com backup: metadados sobre retenção não substituem necessidade de cópias/backups.

Dados Estruturados, Semiestruturados e Não-Estruturados classificam dados conforme sua organização e a facilidade de processamento automatizado.

DADOS ESTRUTURADOS: Organizados em formato rígido, geralmente em tabelas com colunas bem-definidas, tipos de dados específicos, restrições de integridade. Armazenados em Bancos de Dados Relacionais (SGBDR). Exemplos: tabelas SQL (ID, Nome, Email, Data_Nascimento), planilhas Excel, registros de transações bancárias. Características: esquema bem-definido A PRIORI (antes dos dados), possibilidade de consultas SQL precisas, garantias de integridade (constraints, chaves primárias), alto desempenho em buscas e análises estruturadas. Armaze namento eficiente porque estrutura permite compressão. Acessibilidade: fácil busca, filtro, ordenação.

DADOS SEMIESTRUTURADOS: Possuem algum nível de estrutura/marcação sem ser tão rígido quanto estruturados. Marcações (tags, identificadores) fornecem estrutura parcial. Exemplos: XML, JSON, arquivos YAML, e-mails (combinam campo estruturado como "De:", "Para:" com corpo de texto livre). Características: esquema EVOLUCIONÁRIO (pode variar entre registros), estrutura implícita/explícita via marcadores, permite flexibilidade (diferentes campos em diferentes registros), pode ser processado com parsing mas não com SQL simples. Exemplo JSON: `{"nome": "João", "idade": 30, "endereço": {"rua": "Av. X", "número": 100}}` - tem estrutura clara mas cada registro pode ter diferentes campos.

DADOS NÃO-ESTRUTURADOS: Sem formato rígido ou marcação previsível. Exemplos: documentos de texto livre, imagens, vídeos, áudio, logs, e-mails com texto livre, posts em redes sociais, documentos PDF contendo apenas imagens, apresentações. Características: impossível processar com consultas SQL diretas, exigem técnicas especiais (NLP para textos, computer vision para imagens, machine learning para padrões), maior volume potencial, mais difícil indexar/buscar, armazenamento menos eficiente. Valor alto mas exige processamento mais inteligente.

COMBINAÇÕES EM PRÁTICA: (1) Banco de dados relacional = estruturado. (2) API REST retornando JSON = semiestruturado. (3) Data lake contendo estruturados, semiestruturados e não-estruturados = heterogêneo. (4) Documento Word = pode conter texto (não-estruturado) + tabelas (estruturadas).

QUESTÃO CEBRASPE RECORRENTE: "Arquivos .xml, .csv, .json são exemplos de dados estruturados." ERRADO. São semiestruturados (.xml, .json têm marcações; .csv é estruturado mas minimalista sem esquema explícito). Erro comum é classificar qualquer coisa em arquivo como "estruturada". Estruturado exige banco de dados relacional com schema definido.

PROCESSAMENTO: Estruturados: SQL, análise OLAP. Semiestruturados: parsing XML/JSON, ferramentas NoSQL. Não-estruturados: machine learning, NLP, análise manual.
Pegadinhas da banca
.xml e .json são semiestruturados, não estruturados (têm marcações, não esquema rígido de banco).
.csv pode ser considerado estruturado se em contexto de banco de dados, mas é minimalista.
Pensar que semiestruturado significa 'fácil de processar': requer parsing especializado, ainda não é SQL simples.
Confundir 'não-estruturado' com 'inútil': não-estruturado tem valor alto mas exige processamento inteligente (ML, NLP).
Data Lake contém tudo: estruturados, semiestruturados, não-estruturados em volume (Big Data). Data Warehouse é estruturado apenas.

Dados Abertos (Open Data) são dados que podem ser livremente usados, reutilizados e redistribuídos por qualquer pessoa, sem restrições legais, técnicas ou sociais, exceto exigências de atribuição de autoria e compartilhamento sob mesmos termos.

CARACTERÍSTICAS DEFINIDORAS: (1) Disponibilidade e Acesso: dados devem ser integralmente disponibilizados, em formato acessível, com custo não maior que reprodução razoável (idealmente gratuitos). Não devem exigir registro, assinatura de EULA ou acesso restrito. (2) Reutilização e Redistribuição: dados podem ser reutilizados para qualquer propósito, sem restrição de uso ou área de aplicação. Devem poder ser redistribuídos para outros. (3) Participação Universal: sem restrições a pessoas, grupos, ou áreas de atuação. Cientista, empresário, cidadão comum todos podem usar igualmente. (4) Formato Aberto e Não-Proprietário: dados em formato cujo controle não seja exclusivo de nenhuma entidade. CSV, JSON, XML são abertos; Excel proprietário. (5) Primariedade: dados devem ser publicados conforme coletados na fonte, com máxima granularidade, não agregados ou transformados. Permite reanálise com perspectiva diferente.

PRINCÍPIOS DE DADOS ABERTOS (conforme Open Knowledge Foundation e recomendações internacionais): (1) Completos: publicação integral sem censura. (2) Primários: publicados conforme coletados. (3) Oportunos: disponibilizados rapidamente para preservar valor (dados sobre eleição são inúteis 1 ano depois). (4) Acessíveis: fáceis de encontrar e obter. (5) Processáveis: formato que permite processamento automatizado. (6) Não-discriminatórios: acessíveis a qualquer pessoa sem autenticação. (7) Não-proprietários: em formato aberto. (8) Livres de licenças: sob licenças que permitem livre uso (Creative Commons Zero, Open Data Commons).

APLICAÇÕES PRÁTICAS: Dados abertos governamentais (orçamentos públicos, resultados eleitorais, estatísticas de saúde), dados científicos (publicações, datasets de pesquisa), dados corporativos voluntários (empresas que publicam dados para inovação), dados de cidades inteligentes (trânsito em tempo real, clima, qualidade do ar). Transparência, inovação, prestação de contas.

DIFERENÇAS - DADOS ABERTOS VS PRIVADOS VS SENSIVEIS: Dados abertos: sem restrição. Dados privados: acesso restrito (ex: nível de acesso em empresa). Dados sensíveis: pessoais, médicos, financeiros - sujeitos a legislação de privacidade (LGPD, GDPR) mesmo que "abertos" por lei - são abertos mas com proteções.

QUESTÃO CEBRASPE RECORRENTE: Princípio de primariedade exige dados publicados "conforme coletados, com máxima granularidade, não agregados". Banca explora: "dados abertos devem ser agregados para facilitar entendimento?" FALSO - devem ser primários (granulares). Agregação é trabalho do usuário/analista.
Pegadinhas da banca
Confundir 'dados abertos' com 'dados públicos': dados públicos podem ter restrições (acesso por login). Dados abertos são verdadeiramente sem restrição de reuso.
Pensar que 'aberto' significa 'baixa qualidade': dados abertos devem ter qualidade igual à de dados privativos. Falta de qualidade é falha de governança.
Primariedade confundida com 'não processados': primariedade significa granularidade máxima, não que dados devem estar 'sujos'. Limpeza é diferente de agregação/transformação.
Licenças: CC-BY é 'aberta' (permite reuso com atribuição). CC-BY-NC (não-comercial) é restritiva, não é verdadeiramente 'aberta'.
Velocidade: dados sobre eleição publicados 1 ano depois têm pouco valor prático apesar de serem 'abertos' tecnicamente.
